Mission Memory Protocol

A shared brain for AI-driven DevOps. Investigations that survive sessions, tools, and compactions.
Stack — Beads CLI + local Dolt DB Scope — Cross-tool, local-first Storage — ~/.missions/.beads

The Mission Memory Protocol turns any AI tool — Claude Code, Copilot CLI, Pi, Cursor — into a DevOps investigation platform with persistent, cross-tool memory. You work in plain language. Every finding, decision, action, and lesson is recorded in a local database accessible from any AI tool.

Your findings survive sessions. Your lessons survive tools. Your context is never lost again.

Activate
In any AI session, say mission: TOPIC, start a mission, resume mission, or check missions.
§ 01

The Problem

Knowledge that doesn't persist.

DevOps investigations are messy. You run dozens of kubectl, aws, and terraform commands, cross-reference outputs across clusters, services, and accounts — and none of it outlives the session.

After a few deep dives, things break down:

Without mission memory
  • Context window fills with raw command output
  • /compact or /clear wipes the investigation
  • Next day = start from zero
  • Knowledge is trapped in one tool
  • Lessons evaporate between investigations
  • No audit trail of what was checked or decided
With mission memory
  • Findings recorded atomically in Beads; summaries stay in-session
  • DB persists on disk — /compact freely, rehydrate from bd show
  • bd ready / bd show ID restores full context in seconds
  • Same DB accessed from Claude Code, Copilot, Pi, Cursor — any shell
  • bd remember captures cross-mission insights that surface in every future session
  • Timestamped epic → findings / decisions / actions tree, plus threaded comments
“
Core insight
Split the conversation from the knowledge. The AI reasons in its context window — ephemeral. bd remembers on disk — permanent.
§ 02

How It Works

One database, many tools.

The protocol has three concepts:

01 · Mission
An epic in bd, one per investigation. Contains findings, decisions, actions as child issues.
02 · Lesson
A cross-mission insight via bd remember. Surfaces in every future session via bd memories.
03 · Agents
Optional. Dispatch infra-reader or infra-executor as subagents to parallelize.

§ 04

How You Use It

You speak. The AI handles the protocol.

The commands in the table above are what the AI runs. You don’t memorize them, don’t type them, and — on most days — never see them. You say something in plain language and the AI does the bookkeeping. A few patterns cover almost everything:

You say
“mission: ARC runners crashlooping on athena”
AI does
Creates a new epic, labels it, claims it as active.
You say
“update the mission”
AI does
Looks up the active epic, adds findings / decisions / actions from the current conversation, amends the summary as needed.
You say
“resume the mission about arc-runners”
AI does
Searches epics by topic (you don’t need the ID), pulls up findings, decisions, timeline — you’re back in full context in seconds.
You say
“any past missions or lessons about Artifactory tokens?”
AI does
Scans cross-mission memory and epic descriptions, surfaces the relevant lessons and prior investigations.
You say
“new mission — same setup but for the payments team”
AI does
Creates a new epic seeded from the prior one — carries forward relevant findings as starting context, links the two for traceability.
You say
“park it — waiting on platform team”
AI does
Defers the mission, logs the wait-state as a comment so the reason is preserved.
You say
“close it — all fixed”
AI does
Extracts reusable lessons into cross-mission memory, closes the epic with a summary of what happened.

The bd commands you’ll see in the walkthrough are what the AI runs on your behalf — shown for transparency, not as something you need to type.

§ 05

Missions Compound

The payoff is two weeks later.

A single mission is useful. A library of missions is the real win. Old investigations become seed context for new ones — you don’t re-derive what you already figured out. Three patterns come up constantly:

01 · Recall
Ask about a past topic.
Weeks after an investigation, a related question surfaces. Ask about the topic — the AI searches past findings and cross-mission lessons and brings back the fix, the root cause, and the reasoning.
You say
“any past missions about Artifactory tokens?”
02 · Seed
Clone an old mission for new team.
Onboarding another team to the same setup? Point at the old mission — the AI creates a new epic seeded with the prior findings so the new mission starts where the old one ended.
You say
“new mission for the payments team — same arc setup as before”
03 · Branch
Follow a thread downstream.
A related issue surfaces two weeks later — queue stops draining, HPA misbehaves, tokens rotate oddly. The AI opens a linked mission carrying forward the domain knowledge so the new investigation starts with context.
You say
“new arc issue — probably relates to the old mission”
Why it matters
You never typed a single bd command in any of these. Behind the scenes the AI uses bd search, bd memories, bd link to do the work. Your domain knowledge accrues. Future-you (or a teammate in a different tool) starts two weeks ahead.

§ 07

The Protocol

The whole skill is one markdown file.

Everything you’ve read so far — the command conventions, the labels, the lifecycle, the gotchas — lives in a single file at ~/MISSIONS.md. Your AI tool reads it when a mission phrase is detected and follows its instructions. That is the entire skill. No plugins, no binaries beyond bd.

The installer drops it in for you, but if you’d rather copy-paste, here it is — unmodified. Save it to ~/MISSIONS.md and point your AI tool’s system prompt at it (see Installation).

# Mission Memory Protocol You have access to a persistent mission knowledge base via the `bd` CLI (Beads). This is a shared brain across all AI tools — anything recorded here is accessible from Claude Code, Copilot CLI, Pi, Cursor, or any tool that can run bash commands. **Activate when:** the user says "mission: <topic>", "start a mission", "check missions", "search missions", "resume mission", or similar. Otherwise, do not proactively load mission context. **Environment:** `BEADS_DIR` is set in the shell profile (`~/.zshrc`) pointing to `~/.missions/.beads`. Just run `bd` commands directly — no export needed: ```bash bd list bd search "topic" ``` For full `bd` command reference, run `bd prime --full`. This document covers **mission-specific conventions only**. ### Ironclad rules for agents 1. **Missions are global.** `BEADS_DIR` must always point at `~/.missions/.beads`. **Never** create or use a project-local `.beads/` directory. When shelling out from an extension/wrapper, export `BEADS_DIR` explicitly — child processes don't always inherit it from the user's shell profile. 2. **Never use `bd edit`.** It opens `$EDITOR` and will hang an agent. Use **`bd update`** with flags for all programmatic metadata changes: ```bash bd update <id> --title "New title" bd update <id> --priority P2 bd update <id> --add-label foo --remove-label bar bd update <id> --set-labels iam,finding bd update <id> --status closed # equivalent to bd close <id> ``` `bd edit` stays reserved for humans editing descriptions/design notes in their editor. 3. **Never rely on `bd show --current`.** "Current" is a single global pointer per database, which is unsafe when multiple agents/sessions work different missions in parallel. Track the active mission per-session instead (e.g. the pi `/focus` extension), and always pass explicit mission IDs to `bd`. 4. **Titles are prefixed by type.** All child issues of a mission epic start with one of `[finding]`, `[decision]`, `[action]`. This lets `bd children` be skimmed at a glance: anything that's not `[action]` is history. ## Quick Reference | Action | Command | |--------|---------| | **Start mission** | `bd create "Title" --type epic --labels domain1,domain2 -p 1` | | **What's next** | `bd ready` | | **Record finding** | `bd create "What" --parent <epic-id> --labels domain,finding -d "Details"` | | **Record decision** | `bd create "What" --parent <epic-id> --labels domain,decision -d "Reasoning"` | | **Record action** | `bd create "What" --parent <epic-id> --labels domain,action -d "Details"` | | **Record lesson** | `bd remember "Reusable insight" --key domain-slug` | | **List lessons** | `bd memories` | | **Recall lesson** | `bd recall <key>` | | **Forget lesson** | `bd forget <key>` | | **Add comment** | `bd comment <id> "Chronological update"` | | **Amend summary** | `bd note <id> "Description-level amendment"` | | **Search** | `bd search "query" --status all` | | **Search by label** | `bd search "query" --label finding --status all` | | **Search descriptions** | `bd search "query" --desc-contains "detail" --status all` | | **Park mission** | `bd defer <epic-id>` or `bd defer <epic-id> --until="next monday"` | | **Resume mission** | `bd undefer <epic-id>` | | **Close mission** | `bd close <epic-id> --reason "Summary"` | | **Reopen mission** | `bd reopen <epic-id>` | | **Show details** | `bd show <epic-id>` | | **Mission children** | `bd children <epic-id>` | | **Execution timeline** | `bd history <epic-id>` | | **Epic completeness** | `bd epic status <epic-id>` | | **Link missions** | `bd link <id1> <id2> --type relates_to` | | **Stale check** | `bd stale --days 14` | ## Starting a Mission ```bash # Create the mission epic bd create "IAM Permission Boundaries" --type epic --labels iam,security -p 1 # Note the returned ID (e.g., mission-a1b2) — use it as --parent for all child issues # Claim it to mark as actively worked on bd update mission-a1b2 --claim ``` If the user starts with something small that grows, create the epic when scope becomes clear. Retroactively parent earlier findings to the epic. ## Recording Knowledge After each investigation step, capture knowledge **atomically**: **Findings, decisions, actions** — create as child issues of the epic. These are mission-scoped and individually searchable. Title prefix is mandatory; label mirrors the prefix. ```bash # Finding — something discovered during investigation (read-only history) FID=$(bd create "[finding] Dev accounts lack permission boundaries" --parent mission-a1b2 \ --labels iam,finding -d "Checked accounts 111, 222, 333. None have boundary policies." \ --json | jq -r .id) bd close "$FID" # close immediately — it's history, not outstanding work # Decision — a choice made, with reasoning (read-only history) DID=$(bd create "[decision] Use permission boundaries over SCPs" --parent mission-a1b2 \ --labels iam,decision -d "SCPs apply at account level. Boundaries give per-user control." \ --json | jq -r .id) bd close "$DID" # Action — work to do, in progress, or done. STAYS OPEN until actually executed. bd create "[action] Apply boundary to dev accounts" --parent mission-a1b2 \ --labels iam,action -d "Apply rbro-dev-boundary-v1 to accounts 111, 222, 333." # ... later, when done: # bd close mission-a1b2.N ``` **Why close findings/decisions immediately?** They are immutable history. Keeping them "open" makes `bd children <epic>` useless for answering "what's left to do?" — the only legitimate open children should be `[action]` items that haven't been executed yet. **Lessons** — use `bd remember` instead of child issues. Lessons are cross-mission knowledge that should surface in every future session: ```bash # Lesson — reusable knowledge for future missions bd remember "Tags set in org root don't always propagate — verify per-account" \ --key iam-tag-propagation # Domain-prefix keys: iam-*, eks-*, terraform-*, s3-*, etc. bd remember "smec pattern catches smechannel but NOT sme-commons — distinct naming roots" \ --key s3-smec-vs-sme-commons ``` ## Progressive Updates Use **comments** for chronological updates (timestamped, threaded): ```bash bd comment mission-a1b2.1 "Account 444 also missing boundaries" bd comment mission-a1b2.1 "Tried SCP approach — doesn't give per-user control" ``` Use **notes** for amending the description-level summary: ```bash bd note mission-a1b2 "UPDATE: Scope expanded to include prod accounts" ``` ## Searching Past Knowledge ```bash # Cross-mission lessons bd memories bd recall iam-tag-propagation # Mission-scoped knowledge bd search "cross-account" --status all bd search "IAM" --desc-contains "cross-account" --status all # All findings across all missions bd search "findings" --label finding --status all # Everything about arc-runners bd search "arc-runners" --status all --long # Terraform decisions specifically bd search "Decisions" --label decision --desc-contains "terraform" --status all # Find missions by domain bd list --type epic --status all ``` ## Mission Lifecycle **Park** (pause for later): ```bash # Indefinite — no specific resume date bd defer mission-a1b2 bd comment mission-a1b2 "Parked: waiting for security team approval" # Time-based — surfaces automatically after the date bd defer mission-a1b2 --until="next monday" bd comment mission-a1b2 "Parked: waiting for Friday deploy window" ``` **Resume** (continue from any tool): ```bash bd ready # See all actionable work across missions bd show mission-a1b2 # Review where you left off bd children mission-a1b2 # See all findings so far bd history mission-a1b2 # Execution timeline bd undefer mission-a1b2 # Mark as active again ``` **Reopen** (revisit completed mission): ```bash bd reopen mission-a1b2 bd comment mission-a1b2 "Reopened: team asking about tag enforcement approach" # ... add findings ... close again when done ``` ## Landing the Plane "Landing the plane" = capturing the session's knowledge into the mission without necessarily closing the mission. It may be run many times over a mission's lifetime. Closing is a separate, later step. ### Canonical land-the-plane prompt This is the one source of truth. Tools (e.g. the pi `/land` command) should read from this block, not duplicate it. <!-- land-the-plane-prompt:begin --> ``` LAND THE PLANE for the active mission(s): {MISSION_IDS}. Follow the missions skill (which defers to ~/Missions.md): 1. Review this session's work and capture any findings, decisions, or actions that are not yet recorded as `bd` child issues under the mission epic. Use the `[finding]` / `[decision]` / `[action]` title-prefix convention. Close `[finding]` and `[decision]` children immediately; leave `[action]` children open if the work is not yet done. 2. Extract any reusable lessons via `bd remember "insight" --key domain-slug`. 3. File follow-up work as new epics if it is out of scope for the current mission. 4. Do NOT close the mission — only update it with new knowledge from this session. 5. Keep entries atomic, domain-labeled, and avoid duplicating what is already recorded (`bd children <epic-id>` to check first). When finished, give a short confirmation of what you recorded. ``` <!-- land-the-plane-prompt:end --> ### When to actually CLOSE a mission Only close when all of the following are true: 1. All `[action]` children are closed (the work has been executed). 2. Land-the-plane has been run at least once on this session's work. 3. Any reusable knowledge has been captured via `bd remember`. 4. Any out-of-scope work has been filed as a separate epic (and linked via `bd link` if related). Then: ```bash bd epic status mission-a1b2 # sanity-check completeness bd close mission-a1b2 --reason "Summary of what was accomplished" ``` If `bd close` refuses because `[finding]`/`[decision]` children are still open, that means they were created without being immediately closed (old habit). Mass-close them: ```bash bd children mission-a1b2 --json \ | jq -r '.[] | select(.status != "closed") | select(.title | test("^\\[(finding|decision)\\]")) | .id' \ | xargs -n1 bd close ``` or, if you're confident no real action work is hiding, `bd close --force`. ## Mission Hygiene Run periodically (or when resuming work after time away): ```bash bd ready # What can I work on right now? bd stale --days 14 # Anything forgotten? bd memories # Review cross-mission lessons ``` ## Context Recovery If you've lost mission context (after compaction, clear, or session restart): ```bash bd ready # What's actionable right now bd memories # Cross-mission knowledge bd list --type epic --status all # All missions overview ``` Use `bd show <id>`, `bd children <id>`, and `bd history <id>` to recover full context for a specific mission. ## Agents (Optional) If you want to parallelize investigation, dispatch `infra-reader` (read-only) or `infra-executor` (approved changes) as subagents. Not required — running commands directly in the main session works fine for most missions. ## Domain Labels Controlled vocabulary. Prefer these; extend only when a truly new domain appears. Consistency here is what makes `bd search --label <x> --status all` useful across missions. **Infrastructure:** `eks`, `iam`, `acm`, `awx`, `ec2`, `rds`, `s3`, `route53`, `networking`, `docker` **Operations:** `terraform`, `deployment`, `cost`, `security`, `monitoring`, `arc-runners` **Tooling:** `pi`, `claude`, `copilot`, `bd`, `vscode`, `git` **Knowledge type:** `finding`, `decision`, `action` **Work type:** `bug`, `fix`, `test`, `docs`, `ux`, `refactor` **Meta:** `tooling`, `process` ## Security - **Never store secrets** (API keys, passwords, credentials) in mission descriptions - Record *what was done*, not *the credentials used to do it* - Mission data is plaintext in a local Dolt database — filesystem permissions only
That’s the whole skill. Pair it with a working bd install and you’re done.

§ 09

FAQ

The questions that come up every time.
Do I have to memorize any bd commands?
No. You speak in plain language (“start a mission”, “update the mission”, “resume the one about arc-runners”, “any past missions about X?”) and the AI handles every bd call for you. The commands in this article are shown for transparency; the reference lives in ~/MISSIONS.md.
Can I seed a new mission from an old one?
Yes — this is one of the biggest wins. Say something like “new mission, same setup as the arc one from a few weeks ago”. The AI finds the prior mission, creates a new epic carrying forward relevant findings as starting context, and links the two so the relationship is preserved.
What if new work relates to an old mission but isn’t a clone?
Branch it. Say “new arc-runner issue — might relate to the previous mission”. The AI opens a new mission, links it as related, and pulls forward the relevant prior domain knowledge so the new investigation starts with context instead of from scratch.
Can I have multiple missions running at the same time?
Yes — Beads supports any number of concurrent epics. Ask “what missions do I have open?” to list them, or “resume the one about X” to jump back into one.
What happens if I forget to park or close a mission?
Nothing breaks — it stays open. The AI can surface stale missions on request (“anything I’ve forgotten about?”) so nothing gets permanently lost.
Does the AI ever run infrastructure commands directly?
Yes — the main session runs kubectl, aws, terraform, etc. directly. To parallelize or keep raw output out of the context window, dispatch infra-reader or infra-executor as subagents. Optional.
Where is the data stored?
~/.missions/.beads — a local Dolt database. It never leaves your machine. Back it up like any other directory.
How do lessons differ from findings?
Findings are mission-scoped — facts discovered during one investigation. Lessons are cross-mission — reusable insights that should inform every future session. The AI promotes findings into lessons when a mission closes.
